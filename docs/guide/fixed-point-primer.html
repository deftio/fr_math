Fixed-Point Primer

This page explains fixed-point math from scratch. If you’ve seen it before, skim the sM.N notation and radix choice sections — those contain conventions FR_Math actually enforces. The rest is background.

Why fixed-point exists

A CPU’s arithmetic logic unit (ALU) operates on integers. Multiplying two 32-bit integers produces a 64-bit result, adding two 32-bit integers produces a 33-bit result, and so on. The silicon is fast, deterministic, and exists on every chip made since the 1970s.

Fractional numbers do not exist natively in an ALU. If you want to represent 3.14, you have two options:

  1. Floating point — use an IEEE 754 floating-point unit (FPU) or a software float library. The FPU gives you a huge dynamic range (~10−38 to ~1038) for the price of extra hardware and an opaque error model. The software library gives you the same semantics at 10× to 100× the speed cost of hardware float, and pulls in tens of kilobytes of code.
  2. Fixed point — agree with yourself that some of your integer bits represent fractional quantities, and handle the bookkeeping in software.

Option 2 is what FR_Math does.

The core trick

Imagine you have an 8-bit signed integer and you want it to represent temperatures between −64 °C and +64 °C with a resolution of 0.5 °C. You have 7 bits of magnitude, so that’s 128 possible values. Perfect: 128 steps × 0.5 °C = 64 °C of range.

You say “the number in my 8-bit integer is the temperature times 2”, and you handle the ×2 and ÷2 yourself. 25 °C is stored as 50, −13.5 °C is stored as −27. Addition still works the same:

50 + (-27) = 23     (stored integers)
25 + (-13.5) = 11.5 (what they mean)

And 23 ÷ 2 = 11.5. The integer addition was correct — you just had to remember to divide by 2 when you read the number.

This is fixed-point. The “×2” is your scale factor. In FR_Math we always use scale factors that are powers of two, because then “divide by the scale” becomes a right shift, which is free.

Radix notation: sM.N

The scale factor in FR_Math is described by a radix: the number of bits to the right of the binary point. A 32-bit signed integer used as an s11.4 fixed-point number is reserving:

  • 1 bit for the sign (the top bit),
  • 11 bits for the integer portion,
  • 4 bits for the fractional portion,
  • and the remaining 16 bits are headroom — available to absorb intermediate results without overflow.

We write this as s11.4 where:

LetterMeaning
ssigned (2’s complement). Use u for unsigned.
11integer bits in use (not counting sign)
4fractional bits in use — this is the radix

Crucially, the 11 and the 4 describe how many bits the programmer is using, not the register width. The same s11.4 number fits in a 32-bit register (with 16 bits of headroom) or a 16-bit register (with 0 bits of headroom).

The scale factor for a radix-r number is 2r. So:

real_value     = integer_value / (2^radix)
integer_value  = real_value * (2^radix)

Example: 3.5 at radix 4 is 3.5 * 16 = 56, which is 0x38. At radix 8 the same 3.5 is 3.5 * 256 = 896 = 0x380. Same number, different bit pattern, different amount of fractional resolution.

FR_Math never stores the radix alongside the value. It’s your job as the programmer to track which variable is at which radix. The library functions take the radix as an argument so they know how to interpret the bits you pass them.

Operations at a specified radix

Addition and subtraction

a (s11.4) + b (s11.4) = c (s12.4)

Adding two N-bit numbers gives an (N+1)-bit result, so the integer portion grows by one bit on every addition. The fractional portion does not grow. The only requirement is that both operands have the same radix. If they don’t, you have to shift one of them first:

s32 a = /* an s11.4 value */;
s32 b = /* an s9.6  value */;
/* Align b down to radix 4 before adding. */
s32 b_as_s13_4 = b >> (6 - 4);
s32 sum = a + b_as_s13_4;   /* still s?.4 */

FR_Math gives you FR_ADD(x, xr, y, yr, r) which does the alignment for you and produces a result at the requested radix r.

Multiplication

a (sM1.N1) * b (sM2.N2) = c (s(M1+M2).(N1+N2))

Multiplying two numbers gives a result with the sum of the fractional bits on the right of the binary point. So if you multiply two s15.16 numbers, the raw product is an s31.32 value that occupies all 64 bits of an int64_t. You then shift the result right by however many fractional bits you want to keep, and clamp to the destination register width.

FR_Math gives you two spellings:

  • FR_FixMuls(a, b, r) — multiply two same-radix-r numbers and return the r-radix result in an s32. Uses an int64_t intermediate. Truncates — does not clamp.
  • FR_FixMulSat(a, b, r) — same shape, but saturates to FR_OVERFLOW_POS / FR_OVERFLOW_NEG instead of wrapping.

Prefer FR_FixMulSat in any code path where you don’t have a formal proof that the intermediate product stays in range.

Changing radix

FR_CHRDX(value, from_r, to_r) shifts a value between two radixes. If you’re going from a larger fractional width to a smaller one, you lose precision (the low bits are dropped). If you’re going the other way, the new low bits are zeroed in. Either way, the value is conserved as closely as the destination radix allows.

Overflow and saturation

The single biggest trap in fixed-point math is silent integer overflow. A signed 32-bit c = a * b where both are s15.16 can easily overflow a 32-bit intermediate, but C will happily give you wrap-around garbage with no warning.

FR_Math handles this by:

  1. Using int64_t as the intermediate type in every multiply that could plausibly overflow. This is free on any 32-bit or larger CPU and cheap on 16-bit CPUs (sdcc and friends handle it).
  2. Offering FR_FixMulSat and FR_FixAddSat variants that clamp to INT32_MAX / INT32_MIN (aliased as FR_OVERFLOW_POS / FR_OVERFLOW_NEG) instead of wrapping.
  3. Returning a dedicated sentinel FR_DOMAIN_ERROR (INT32_MIN) from functions with invalid inputs — e.g. FR_sqrt(-1), FR_log2(0), FR_atan2(0, 0).

You should check for these sentinels in debug builds. In release builds on well-bounded inputs, the sentinel paths simply never trigger.

Choosing a radix

The best radix for a given variable depends on three things:

  1. Maximum absolute value the variable ever needs to hold. Take ceil(log2(max)) to get the number of integer bits you need.
  2. Required precision (smallest meaningful step). Take ceil(-log2(step)) to get the number of fractional bits you need.
  3. Register width available on your target CPU.

Worked example: you’re building a PID loop for a motor controller. The error signal has a maximum of ±512 ticks, and you need to resolve ±0.01 ticks to avoid limit-cycle oscillation.

  • Integer bits: ceil(log2(512)) = 9
  • Fractional bits: ceil(-log2(0.01)) = 7
  • Sign bit: 1

Total: 9 + 7 + 1 = 17 bits. Doesn’t fit in 16 bits. You need an s32 and you could pick s9.7 (17 bits used, 15 headroom) or go wider — e.g. s9.22 uses all 32 bits for real precision.

FR_Math encourages using just enough rather than always picking s15.16 out of habit. A narrower radix leaves more headroom for intermediate products and enables tighter packing of state in RAM.

The I2FR / FR2I macros convert between plain integers and fixed-point bit patterns:

  • I2FR(i, r) = (i) << (r) — integer to radix-r fixed-point.
  • FR_FR2I(x, r) = (x) >> (r) — radix-r fixed-point to integer (truncates toward −∞ for negative values because of C shift semantics; use FR_FR2Iround for round-to-nearest).

Angle representations

FR_Math supports three angle representations and converts cleanly between them:

  1. Integer degrees — s16, 0 ≤ θ < 360. Used by the classic FR_Sin / FR_Cos API.
  2. Radians at a programmer-chosen radix — used by fr_sin / fr_cos / fr_tan.
  3. BAM (Binary Angular Measure) — a u16 where the full circle maps to [0, 65536). BAM is the native representation of the internal trig table: the top 2 bits select the quadrant, the next 7 bits index the table, and the bottom 7 bits drive linear interpolation. BAM has no quantisation error at the wraparound, which is very handy for phase accumulators in audio / LFO work.

Convert between them using macros: FR_DEG2BAM, FR_BAM2DEG, FR_RAD2BAM, FR_BAM2RAD, FR_DEG2RAD, FR_RAD2DEG. See the API reference for the full list.
