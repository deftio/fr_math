
# FR_Math TDD Characterization Report

> Generated by `tests/test_tdd.cpp`. This is a measurement suite, not a pass/fail suite.
> All numbers below are *what the library actually does*, compared to libm `double` references.

## 0. Platform & Type Sizes

| Type | Size (bytes) |
|---|---:|
| s8  | 1 |
| u8  | 1 |
| s16 | 2 |
| u16 | 2 |
| s32 | 8 |
| u32 | 8 |
| int | 4 |
| long | 8 |

> **WARNING**: `s32` is 8 bytes, not 4. The library `typedef`s `s32` as `signed long`, which is 64-bit on LP64 platforms (Linux, macOS x86_64/arm64). All overflow/saturation behavior is tested with 64-bit accumulators here, which **masks bugs that would fire on a real 32-bit MCU**.

### 11.3 New findings not in dev/fixes.md

- `FR_DEG2RAD` and `FR_RAD2DEG` macro **bodies are swapped relative to their names**.
- `FR_NUM` ignores its fractional argument silently.
- `FR_SQUARE` macro is uncompilable (missing close paren). Dead code.
- `FR_atan` is declared in `FR_math.h` line 292 but **not implemented** in `FR_math.c`.
- `FR_atan2` is implemented in `FR_math.c` but **missing from the public header**.
- `FR_pow2` is broken on **negative fractional exponents** (the negative-flr path), in addition to FR_log2 being broken.
- `s32` is `signed long`, which is 64-bit on LP64 platforms. **All host-side overflow tests are silently using 64-bit arithmetic** and cannot expose 32-bit MCU bugs in `FR_FixAddSat`, `FR_FixMulSat`, or any callsite that depends on 32-bit wrap.
- `FR_TanI`'s `if (270 == deg)` branch is **dead code** after the `% 360` then `[-180,180]` reduction. (Confirmed via gcov.)
- `FR_printNumD`/`F`/`H` return `FR_S_OK` (which equals 0), not the byte count promised by the `int` return type and the `int` declaration in the header.


### 11.4 Coverage

- `FR_math.c`: this TDD suite alone reaches **98.2%** of lines (218/222). The 4 uncovered lines are: 3 deeper saturation branches inside `FR_FixMulSat` (lines 76, 81, 86) that need very specific intermediate overflow patterns, plus 1 dead code path (`FR_TanI` line 209: `if (270 == deg)` is unreachable after the `% 360` reduction).
- `FR_math_2D.cpp`: **98.0%** (99/101). The 2 uncovered lines are `inv()` failure returns inside `inv(FR_Matrix2D_CPT*)` for malformed input.
- `FR_math_2D.h`: **100%** (4/4 inline lines).
- Every macro in `FR_math.h` is exercised at least once in section 2 / 3.
- Run `gcov -o build build/test_tdd_FR_math.gcno` after `make test-tdd` to confirm.


### 11.5 Recommendation

Treat the following as **release blockers** for any production use:
1. `FR_log2`, `FR_ln`, `FR_log10` — fundamentally wrong algorithm.
2. `FR_pow2` for negative fractional inputs — returns reciprocal-wrong values.
3. `FR_atan2` — wrong return contract.
4. `FR_Tan` — broken above 45° due to s16 truncation of s32 intermediates.
5. `FR_FixMulSat` — wrong multiply algorithm.
6. `FR_DEG2RAD` / `FR_RAD2DEG` — swapped bodies.
7. `FR_NUM` — ignores fraction argument.
8. `FR_SQUARE` — uncompilable.
9. `FR_printNumD` / `FR_printNumF` — wrong fractions and INT_MIN/MAX overflow.
10. `s32` typedef — should be `int32_t` from `<stdint.h>`, not `signed long`, otherwise host tests cannot validate 32-bit MCU behavior.

The following are **fine as-is**:
- All header constants.
- `FR_CosI`, `FR_SinI`, `FR_TanI` (integer degrees), `FR_Cos`, `FR_Sin`.
- `FR_acos`, `FR_asin`.
- `FR_FixMuls`.
- The whole `FR_Matrix2D_CPT` 2D matrix class.
- Most basic macros (`FR_ABS`, `FR_SGN`, `I2FR`, `FR2I`, `FR_INT`, `FR_CHRDX`, `FR_FRAC`, `FR_FRACS`, `FR_ADD`, `FR_SUB`, `FR_INTERP`, `FR_INTERPI`, `FR_FLOOR`, `FR_CEIL`, `FR_ISPOW2`, `FR_SWAP_BYTES`).
